商品规格组件

SPU 和 SKU

  • SPU（Standard Product Unit）：标准化产品单元。是商品信息聚合的最小单位，是一组可复用、易检索的标准化信息的集合，该集合描述了一个产品的特性。通俗点讲，属性值、特性相同的商品就可以称为一个 SPU。

  • SKU（Stock Keeping Unit）库存量单位，即库存进出计量的单位， 可以是以件、盒、托盘等为单位。SKU 是物理上不可分割的最小存货单元，在使用时要根据不同业态，不同管理模式来处理。

总结一下：SPU 代表一类商品，例如 iPhone6；SKU 代表该商品可选规格的任意组合，是具体的一个商品。

禁用思路分析

推荐演示商品：http://localhost:3000/#/goods/1369155859933827074，大致了解禁用效果的整体思路即可。

  1. 根据后台返回的 skus 数据得到有效 sku 组合（过滤掉库存为 0 的）。

  2. 根据有效的 sku 组合得到所有的子集集合。

  3. 根据子集集合组合成一个路径字典（对象）。

  4. 在组件初始化的时候去判断每个规格是否可点击，判断依据是：判断每一个按钮的 name 在字典中能不能找到。

  5. 在点击规格的时候去判断其他规格是否可点击，判断依据是：用已选中的和其他按钮一一进行组合，判断组合后的结果在字典中能不能找到。

一个 Bug

  1. 进入 http://localhost:3000/#/goods/1379052170040578049，默认 20cm 被禁用，没有问题。

  2. 随便点击某个一级类目，再点击浏览器返回，发现 20cm 禁用的效果丢失了。

  3. 分析：返回时，渲染 views/goods/index.vue 组件，此时 Pinia 中获取的 info.categories 是有数据的，所以会直接渲染 GoodsSku 组件。

  4. 执行 GoodsSku 组件中的代码（setup 和 template），该禁用禁用。

  5. 而 views/goods/index.vue 中会再次发起请求，从后端拿到最新的数据，会覆盖原数据，注意此时会把原来前端计算好的 disabled 的覆盖掉，可以在 goods/index.vue 的模板中输出 {{ info.specs[1].values[1] }} 证明。

  6. 此时数据会再次渲染 goods/index.vue 模板，渲染 goods/components/goods-sku.vue 模板，但是！goods-sku.vue setup 中的逻辑则不会再被执行了。

  7. 解决方式 1：保证 goods-sku.vue 重新执行 setup 中的代码。

  1. 解决方式 2：请求新数据前或组件销毁前，删除 Pinia 中的旧数据，保证 goods-sku.vue 模板中的渲染只执行 1 次（当然这一次肯定也会经过 setup 中的逻辑处理）。
