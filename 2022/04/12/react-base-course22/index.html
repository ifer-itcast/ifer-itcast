评论列表

Redux MobX
有严格的工作流程，需要写大了的模板代码 无模板代码，非常简单
需要保证数据不可变 数据是响应式的，可以直接修改数据（Proxy）
需要中间件处理异步 可以直接处理异步
Redux 约束强，更适合大型多人协作开发 Mobx 适合简单，规模不大的应用

核心概念

observable 定义一个存储 state 的可追踪字段（Proxy）

action 将一个方法标记为可以修改 state 的 action

computed 标记一个可以由 state 派生出新值并且缓存其输出的计算属性

this 指向问题

computed

makeAutoObservalble

像加强版的 makeObservable，默认情况下它将推断所有的属性，推断规则如下。

所有的属性都成为 observable；所有的方法都成为 action；所有的 get 都成为 computed。

可以通过参数二排除不需要被观察的属性和方法，空对象表示不排除。

可以通过参数三 autoBind 绑定 this 指向。

监听属性

一上来会运行一次。

autorun 接收一个函数作为参数，每当该函数所观察的值发生变化时，它都应该执行。

MobX 会自动收集并订阅所有的可观察属性，一旦有改变发生，autorun 将会再次触发。

src/store/Counter.ts

reaction 类似于 autorun，但可以让你更加精细的控制要跟踪的可观察对象，它接收两个函数作为参数。

参数 1：data 函数，其返回值将会作为第二个函数输入。
参数 2：回调函数。

与 autorun 不同，reaction 在初始化时不会自动运行。

MobX 处理异步

异步代码在 MobX 中几乎不需要任何特殊处理，但如果可观察对象的修改不是直接在 action 函数中，控制台会报警告（可以关闭，但不推荐）。

runInAction: 这样就不必拆分两个函数啦

MobX 模块化

项目大了之后，不能将所有的状态和方法都放到一个 Store 中，可以根据业务模块定义多个 Store，通过一个根 Store 统一管理所有的 Store。
